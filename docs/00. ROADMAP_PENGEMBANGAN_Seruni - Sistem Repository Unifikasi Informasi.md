## 0. CARA MENGGUNAKAN ROADMAP INI

Roadmap ini dibagi 3 bagian wajib yang harus dijalankan **berurutan**:

1. **BAGIAN 1 — Urutan Baca** → Agent membaca dokumen dari nomor 1 sampai terakhir **SEBELUM menulis satu baris kode**. Tujuannya: pahami peta sistem, skema kanonik, dan aturan sebelum membangun.
2. **BAGIAN 2 — Urutan Eksekusi (Build Phases)** → Setelah paham, bangun per phase sesuai urutan dependensi (bukan urutan halaman dokumen).
3. **BAGIAN 3 — Aturan Hindari Konflik** → Aturan mutlak yang mencegah tabrakan skema, duplikasi, dan event tidak konsisten.

> **Prinsip emas:** _"Satu Input, Banyak Dampak"_ — tidak ada duplikasi fakta antar modul; semua efek menyebar lewat Event Propagation Layer. Baca [`WORKFLOW_SISTEM_KESELURUHAN.md` §0](WORKFLOW_SISTEM_KESELURUHAN.md:8) sebelum memulai BAGIAN 2.

## BAGIAN 2 — URUTAN EKSEKUSI (Build Phases)

Eksekusi per phase **berurutan**. Setiap phase menyatakan: tujuan, file yang WAJIB sudah dibaca, apa yang dibangun, kriteria selesai, dan catatan anti-konflik. Urutan ini mengikuti **§C11 (migrasi dependensi)**, BUKAN urutan halaman dokumen.

### Phase 0 — Scaffold Monorepo

- **Sudah baca:** #1 Pedoman (§2.2 struktur, §2.3 konvensi).
- **Bangun:** Struktur `desaku-monorepo/` (`apps/web`, `apps/worker`, `apps/wa-bot`, `packages/db`, `packages/ui`, `packages/config`, `packages/validators`, `packages/events`, `packages/utils`), `turbo.json`, `package.json` root, ESLint+Prettier (strict mode, `noAny:true`).
- **Selesai jika:** `pnpm install` & `turbo build` berjalan; TypeScript strict aktif di semua package.
- **Anti-konflik:** Ikuti konvensi commit Conventional Commits & branch `feat/<scope>` (§2.3). Jangan hardcode env — pakai `packages/config`.

### Phase 1 — Fundasi Database & Migrasi (§C11 langkah 1–29)

- **Sudah baca:** #3 SKEMA (§C1 konvensi, **§C11 urutan migrasi**), #6 Audit (F-01 skema §C25 sudah sinkron).
- **Bangun:** File migrasi Drizzle benama `0001_tenants.sql` … mengikuti **persis 29 langkah §C11** (tenants → penduduk → wilayah_batas → … → skema zero-hardcode terakhir di langkah 29). Sertakan seed tabel referensi (`ref_agama`, `ref_pendidikan`, dll) & `bidang_kegiatan`/`sub_bidang`/`rekening_anggaran`.
- **Selesai jika:** Migrasi jalan tanpa error FK; RLS `tenant_isolation` aktif di semua tabel domain; `UNIQUE(tenant_id, nik)` & index penting ada.
- **Anti-konflik:** **JANGAN** menebak urutan tabel — pakai §C11. **JANGAN** edit tabel turunan (`idm_skor_cache`, `dashboard_agregat`, `pades_pendapatan`, `aset_penyusutan`, `usulan_kegiatan_draft_otomatis`) lewat migrasi isi manual — hanya struktur tabel, diisi worker (§C10).

### Phase 2 — Core Registry + Event Propagation Layer (Fondasi Integrasi)

## BAGIAN 3 — ATURAN HINDARI KONFLIK (Wajib Mutlak)

Aturan ini mencegah tabrakan antar agent/sesi. Langgar = konflik skema/duplikasi/data.

1. **Single Source of Truth per Fakta** — identitas warga hanya di `penduduk`; objek pajak hanya di `objek_pajak`. Tidak ada duplikasi antar modul (Prinsip 1, §3).
2. **Event Propagation Layer wajib** — semua perubahan fakta mentah → `domain_events`; efek turunan dihitung worker, **tidak pernah input manual** (Prinsip 2).
3. **Fakta mentah vs turunan dipisah tegas** — tabel turunan (`idm_skor_cache`, `dashboard_agregat`, `pades_pendapatan`, `aset_penyusutan`, `usulan_kegiatan_draft_otomatis`) **HANYA ditulis worker**; tidak ada endpoint admin edit langsung (§C10, Prinsip 3).
4. **Urutan migrasi = §C11, bukan urutan halaman** — jalankan `CREATE TABLE` per 29 langkah dependensi (§C11). Jangan tebak FK.
5. **Nama event seragam** — gunakan event kanonik dari `Sistem_*.md` (sudah diselaraskan ke §C7.1 & Workflow). Contoh: `potensi.wisata.terbit`, `bencana.alert.dikirim`, `informasi.berita.terbit`, `perangkat.berubah`. Jangan buat nama baru.
6. **Penomoran fitur F1–F10** — konsisten di semua dokumen (F0 tidak ada). Jangan pakai "F0".
7. **Zero-Hardcode di Frontend** — tidak ada teks/warna/menu/section tetap di kode; semua dari `tenant_theme_config`/`site_content_blocks`/`site_navigation`/`feature_flags`/`i18n_strings` (Prinsip 6, §7).
8. **Multi-Tenancy** — setiap tabel domain punya `tenant_id NOT NULL` + RLS `tenant_isolation`; middleware validasi `subdomain`→`tenant_id` (§9.1).
9. **Privasi berlapis** — NIK penuh & data kesehatan individu & `bidang_tanah.lokasi_geom` warga **tidak pernah** tampil publik (§9.3, §7.8.1).
10. **Tidak ada aksi kritikal tanpa persetujuan manusia** — draft otomatis (usulan, aset, ekspor) selalu `menunggu_review`, **tidak auto-approve** (Prinsip 7).
11. **Worker idempoten** — selalu `ON CONFLICT DO UPDATE` untuk tabel cache; set `processed_at` setelah proses (§8.2).
12. **Precedence dokumen** — jika bentrok: `Seruni - Sistem Repository Unifikasi Informasi_MASTER_SPEC_FINAL.md` > `PEDOMAN_MONOREPO_Seruni - Sistem Repository Unifikasi Informasi.md` > `SKEMA_DATABASE_ERD.md` > `Sistem_*.md` > lainnya. Workflow & Audit mengikuti master spec.
13. **Jangan hapus/ubah file inti tanpa konfirmasi** — 33 file inti (lihat daftar di bawah) adalah rujukan build; jangan edit kecuali untuk sinkronisasi terselaraskan (selalu perbaiki skema kanonik dulu).

**33 File inti (jangan dihapus):** `Seruni - Sistem Repository Unifikasi Informasi_MASTER_SPEC_FINAL.md`, `PEDOMAN_MONOREPO_Seruni - Sistem Repository Unifikasi Informasi.md`, `SKEMA_DATABASE_ERD.md`, `DESAIN_FRONTEND_KANTOR_DESA_VIRTUAL.md`, `WORKFLOW_SISTEM_KESELURUHAN.md`, `AUDIT_INTEGRASI_Seruni - Sistem Repository Unifikasi Informasi.md`, `JENIS_SURAT_KONSOLIDASI.md`, `PRD_KANTOR_DESA_VIRTUAL.md`, dan ke-24 `Sistem_*.md`.
